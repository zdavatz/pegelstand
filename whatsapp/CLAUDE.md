# WhatsApp integration & Pump TsĂĽri Welcome

Guidance for the `whatsapp/` scripts and the `sync-contacts` / `welcome` flow. This file loads only when working under `whatsapp/`.

Note: the `welcome` flow also spans `src/` Rust (`main.rs`, `gmail.rs`, `onedrive.rs`, `sync_contacts.rs`, `google_sheets.rs`) â€” documented here for reference. The safety rule "never commit real subscriber numbers" lives in the root `CLAUDE.md` so it is always loaded, including when editing those src/ files.

## WhatsApp Integration

- Located in `whatsapp/` directory â€” standalone Node.js scripts using Baileys (`@whiskeysockets/baileys` v7)
- **Requires Node.js â‰Ą 22** (Baileys v7 segfaults on Node 20.2; nvm path `/home/zeno/.nvm/versions/node/v22.22.2/bin/node`)
- `send.mjs` â€” send image to a WhatsApp group JID with optional caption; uses `process.exit(0)` after send to avoid close-handler hangs
- `send-doc.mjs` â€” send any file type (PDF, CSV, log, image, ...) to a JID *or* a plain phone number (`41787496544` is auto-suffixed to `@s.whatsapp.net`). Picks `image:` for png/jpg/jpeg, `document:` for everything else with a mime-type lookup table. Uses a 5-minute connection timeout (room for QR scan) and a 10-second post-send delay so async `saveCreds()` finishes writing `auth/creds.json` before node exits â€” without the delay, the next send needs a fresh QR.
- `list-groups.mjs` â€” list all groups with their JIDs
- `login.mjs` â€” dedicated login flow; auto-wipes stale sessions on 401 (loggedOut) and auto-reconnects on 515 (restartRequired, fires after first QR scan). Uses a `done` flag to prevent open/close race from reporting a spurious error when open fires moments before the socket closes.
- `login-qr.mjs` â€” same login flow but renders the QR additionally as a PNG (`/tmp/wa-login-qr.png`, via the `qrcode` package) and opens it in a window (`feh --auto-zoom`, fallback `xdg-open`). Use when the terminal ASCII QR is hard to scan. `--force` wipes the old session first. The send path (`check-and-send.mjs`) hangs silently waiting for a QR if `whatsapp/auth/creds.json` is missing â€” check that file exists before any send and re-login via this script if not. **Auto-reconnect loop**: WhatsApp only hands out ~6 QR refs per connection, then closes with `408` "QR refs attempts ended"; `main()` now loops (reconnect for a fresh QR batch, up to 10 min total; wipe+restart on 401/403) instead of dying â€” so a remote user relaying the ASCII QR always has a live QR. **âš ď¸Ź Do NOT abuse this into many scans:** too many device-link attempts trips a WhatsApp account block (phone popup "VerknĂĽpfen neuer GerĂ¤te derzeit nicht mĂ¶glich â€” Bitte versuche es spĂ¤ter erneut"), which then fails BOTH QR and pairing until you wait hours. One clean scan; if it fails with that popup, stop and wait.
- `login-pair.mjs` â€” **experimental/unreliable** login via 8-char pairing code instead of QR (for remote/headless). Failed every time on 2026-07-10: 4Ă— against Baileys `7.0.0-rc13` (phone "GerĂ¤t konnte nicht hinzugefĂĽgt werden"; socket stuck `registered:false`; close code 408 "attempts ended"), and retested on **stable Baileys 6.7.23 â†’ same 408**, so it is NOT an RC-version bug â€” the pairing-code handshake is just broken for this account. **Prefer `login-qr.mjs`**; keep this only in case a future Baileys release fixes it.
- `leave-group.mjs` â€” supports comma-separated JIDs for bulk-leave
- `add-to-group.mjs` â€” add one or more numbers to a group (`groupParticipantsUpdate(..., "add")`); logs status 200 added / 403 privacy-blocked. Does *not* itself send an invite link.
- `send-group-mention.mjs` â€” post a **text message with @-mentions** to a group. Usage: `node send-group-mention.mjs <job.json>`, job = `{groupJid, text, mentions:[jidâ€¦]}`. The `text` must already contain the `@<number>` tokens (no `+`); `mentions` carries the JIDs Baileys writes into `contextInfo.mentionedJid`. Before sending it fetches `groupMetadata` and **warns for every mentioned JID that is not a group member** â€” for those WhatsApp renders the raw phone number to the whole group instead of a name. Driven by `welcome --announce` (below); 10s post-send delay for the `saveCreds()` flush, same as `send-doc.mjs`.
- `send-group-invite.mjs` â€” send a group **invite link** to one or more numbers. Fetches `sock.groupInviteCode(groupJid)` â†’ `https://chat.whatsapp.com/<code>` and DMs it with a short note. Use when a direct add is blocked (403). Usage: `node send-group-invite.mjs <groupJid> <number> [<number>...]`.
- Session auth stored in `whatsapp/auth/` (excluded from git)
- First run requires QR code scan (WhatsApp â†’ Linked Devices)
- Uses `fetchLatestBaileysVersion()` + `makeCacheableSignalKeyStore()` for stable connection
- Rust binary calls Node.js script as subprocess via `std::process::Command`; `find_node()` searches nvm, Homebrew, and system paths

## Pump TsĂĽri Welcome (`sync-contacts` / `welcome` alias)

End-to-end flow: Google Form (responses sheet) â†’ SQLite â†’ Baileys WhatsApp send, with a **Gmail-API e-mail fallback** for signups not reachable on WhatsApp (see `src/gmail.rs` below).

- **Auth**: service-account JSON at `whatsapp/google-sa.json` (gitignored). `src/google_sheets.rs` signs an RS256 JWT (`jsonwebtoken` crate), exchanges for an access token, then calls `spreadsheets.values.get`. No public sharing of the sheet â€” only the SA email needs Viewer access.
- **Sheet resolution**: input is a URL OR ID; `parse_sheet_id_and_gid()` extracts both. `resolve_sheet_title()` fetches metadata to map `gid` â†’ sheet tab title (e.g. "Antwort", "Formularantworten 1"), then `fetch_values()` reads `<title>!A:Z`.
- **Presets**: `Commands::SyncContacts` takes an optional positional `variant`. Three `WelcomePreset` constants in `main.rs` bundle the per-variant defaults â€” sheet URL, DB filename, welcome text, default-image flag, and column letters. Each preset's defaults are applied via `unwrap_or_else` after clap parsing, so individual flags (`--sheet`, `--welcome`, `--mobile-col`, `--db`, â€¦) still override on a per-field basis. Current presets:
  - `pumper` (default) â€” Pump-TsĂĽri signup, `contacts.db`, columns C/J/D, image on.
  - `pp` (power pumper, one-minute achievement) â€” different sheet, `contacts_pp.db`, columns D/C/B, image off, Twint/cap message.
  - `build` (build & pump event) â€” `contacts_build.db`, columns E/C/D (mobile/first/last), image off, English text-only welcome ("Welcome to the build and pump event {first}."). The signup sheet must be shared with the service-account email (`pegelstand@pegelstand.iam.gserviceaccount.com`) as Viewer or reads 403. To message only one new entry (e.g. just today's signup), `--mark-existing` to mark the whole pending set as greeted, `DELETE` that one row from `contacts`, then a normal run sends to it alone.
- **Storage**: `whatsapp/contacts*.db` (SQLite via `rusqlite` with `bundled` feature â€” no system dep). `db_path(name)` and `open_db(name)` take the filename so multiple variants stay isolated. Two tables per DB:
  - `submissions` â€” full raw archive. PK = `row_index` (1-based, header is row 1). `data` column is a JSON object keyed by header name (source of truth for any schema evolution). Plus one TEXT column per sheet header, added on the fly via `ALTER TABLE` when a new header appears, and backfilled from JSON for existing rows so old data is immediately queryable without `json_extract`. Sanitization: lowercase, non-alnum â†’ `_`, capped at 50 chars, deduped on collision.
  - `contacts` â€” registry of WhatsApp-confirmed recipients. PK = `jid`. FK `row_index` â†’ `submissions(row_index)`. Inserted with `INSERT OR IGNORE` only **after** Baileys confirms the send.
- **Phone normalization** (`normalize_phone()` in `sync_contacts.rs`):
  1. Token-extraction: scans the cell for `+digits` or bare-digit runs (â‰Ą7 digits) bounded by separators (space/dash/parens/dot/comma/slash). Letters and multi-byte punctuation terminate a token. Multiple tokens per cell are kept as candidates.
  2. Multi-`+` rule: if multiple `+`-prefixed candidates and the lowercase string contains "whatsapp", pick the candidate nearest (byte distance) to the word. Handles `+41 77 902 18 93 (Whatsapp: +48696905840)` â†’ `+48696905840`.
  3. CC normalization: strip `00` â†’ `+`; leading `0` â†’ `+<default_cc>`; bare 9-digit Swiss starting with `7` â†’ `+41â€¦` heuristic.
  4. Length sanity: 10â€“15 digits; `+41` numbers must be exactly 11 digits after `+`.
- **Node helper**: `whatsapp/check-and-send.mjs` â€” reads job JSON `{contacts, welcome, imagePath, groupJid}` from a temp file, connects via Baileys, calls `sock.onWhatsApp(jid)` per contact, sends image+caption (or text only if no `imagePath`), writes results JSON back. 1.5s pause between sends, 10s post-send delay for `creds.json` flush (same pattern as `send-doc.mjs`). If `groupJid` is set it also `groupParticipantsUpdate(..., "add")`s each recipient; on status **403** (recipient's privacy settings block a direct add) it automatically fetches the group invite code (`sock.groupInviteCode`, cached once per run) and DMs the person the invite link (`https://chat.whatsapp.com/<code>`), setting `invited: true` in the result. Log line: `â†’ privacy blocks add; invite link sent`.
  - **Delivery watching** (`WA_WATCH_DELIVERY=1`, set by pegelstand's `--watch-delivery`): the helper tracks each sent message's `key.id`, subscribes to `messages.update`, and records the highest ack seen (`SERVER_ACK=2` accepted-by-server / `DELIVERY_ACK=3` reached-device / `READ=4`). The post-send keepalive stretches from 10s to **50s** (unless `WA_KEEPALIVE_MS` is set explicitly) so the recipient's device has time to ack. At exit it re-writes `out.json` with `delivery` + `deliveryStatus` per result and prints a `Delivery summary:`. This is the built-in, **non-duplicating** way to confirm delivery â€” the same logic `diag-delivery.mjs` uses standalone, but folded into the normal welcome send (no second message). Rust reads it back via the optional `delivery`/`deliveryStatus` fields on `ResultEntry` and prints its own `Zustellung (Delivery-Quittungen):` summary.
- **PNG generation**: by re-invoking `std::env::current_exe()` with `svg --start â€¦ --end â€¦ --png` rather than duplicating chart code. Expected output path: `png/zurichsee_{start}_{end}.png`. Skipped if preset has `default_image: false` OR `--no-image` passed (effective = `no_image || !preset.default_image`).
- **CLI flags**: positional `variant`, then `--sheet`, `--mobile-col`, `--first-col`, `--last-col`, `--welcome`, `--db` (all `Option<String>` â€” preset supplies defaults); plus `--cc 41`, `--days 3`, `--no-image`, `--dry-run`, `--mark-existing`, `--regen-docs`, `--regen-rows "132,135"`, `--with-whatsapp`, `--watch-delivery` (`--force-email` still parses but is a no-op â€” e-mail is the default since 18.07.2026).
- **`--watch-delivery`**: after sending, keep the Baileys socket open ~50s and watch the delivery receipts so the real per-contact ack level is logged (`SERVER_ACK` vs `DELIVERY_ACK`) â€” confirms delivery **without** a duplicate send. Sets `WA_WATCH_DELIVERY=1` on the Node helper; the ack is read back from `out.json` and printed as `Zustellung (Delivery-Quittungen):`. Use it whenever you'd otherwise be tempted to re-send "just to check" (which duplicates). Related: [[feedback-whatsapp-send-no-kill]] and `whatsapp/diag-delivery.mjs` (the standalone one-number probe).
- **`--dry-run`** writes submissions (passive archive) but skips PNG generation, Node helper, and contacts insert.
- **`--mark-existing`** writes the current pending set straight into `contacts` without sending â€” one-shot backfill for an existing form population so the next run only messages genuine new submissions.
- **`--regen-docs`** (pp only) regenerates OneDrive MĂĽtze documents for already-registered `contacts` **without** any WhatsApp send â€” recovery path when the OneDrive login failed on the original run, or to re-create docs. Queries `contacts` (first_name/last_name/row_index), looks up the address from the live sheet row, parses + replaces placeholders, uploads. `--regen-rows "132,135"` restricts to specific 1-based row indices; empty = all registered contacts. Branch returns early before the send/PNG path.
